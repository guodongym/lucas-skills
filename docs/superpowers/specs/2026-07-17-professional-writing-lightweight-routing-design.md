# professional-writing 轻量叙事路由设计

- 日期：2026-07-17
- 状态：已批准并实现，待真实任务验证
- 基线：`main@2e4e7a6`，即已通过人工验收的 beta
- 历史参考：`backup/professional-writing-real-feedback-20260717@65a0f6e`

## 1. 问题

一份真实的系统演进汇报最初按建设时间线展开，读者需要读到后面才知道系统能做什么、为什么这样做、如何运行以及当前状态。现有 skill 已要求“按读者认知路径组织”，但没有明确区分“介绍系统”与“复盘过程”这两种首要阅读任务，模型容易沿素材顺序写成历史流水账。

目前只有这一份真实文档暴露该问题。证据不足以支持四角色写作拓扑、新事实锁层或大规模合成评测；已验收 beta 应继续作为运行时基线。

## 2. 目标

仅增强叙事地图的条件式路由：

- 系统介绍先回答能力与当前价值，再回答动机、构建或运行方式、当前状态；历史演进仅在有助于解释取舍时后置展开。
- 以过程复盘为目标的文档继续按影响、原因或过程、修复与行动组织。
- 普通总结和进展汇报继续按结论或当前状态、证据与影响、下一步组织。

该规则约束的是读者问题的优先级，不规定章节数量、标题名称或固定蓝图。作者可以按材料和篇幅合并、省略或改写问题。

## 3. 非目标

- 不实现 §13 的架构师、fresh writer、独立 reviewer 或源盲 reader 拓扑。
- 不新增事实锁字段、角色与决策权清单、固定首屏字段或文档模板。
- 不修改 `references/`、`evals/`、fixtures 或现有事实保真合同。
- 不把单个真实样本扩张成所有总结都必须采用的章节顺序。
- 不宣称新规则已经通过真实任务验证。

## 4. 最小运行时改动

只修改 `skills/professional-writing/SKILL.md` 的“4 画叙事地图”。在现有“按读者追问排序”规则中加入以下语义，不增加新的流程阶段或检查表：

> 先判断文档的首要阅读任务。系统介绍优先回答“能做什么、为什么这样做、如何构建或运行、当前处于什么状态”，再按需要补充历史演进；过程复盘按影响、原因或过程、修复与行动组织；普通总结和进展汇报按结论或当前状态、证据与影响、下一步组织。这里约束的是问题优先级，不是固定章节模板，可按材料合并或省略。

最终措辞可在不改变上述语义的前提下压缩，以保持 runtime 简洁。

## 5. 验收

合入前验证：

1. `SKILL.md` 仅增加上述条件式路由，不引入 §13 拓扑或新的清单。
2. 系统介绍、过程复盘、普通总结三条路径均被显式区分；教程和其他既有类型不受影响。
3. `quick_validate` 通过。
4. 仓库现有单元测试全部通过，`git diff --check` 无错误。

合入后的产品验证：在 3～5 个真实写作任务中记录是否再次出现“系统机制晚于历史”或结构僵化。只有同类问题在至少 2 份真实文档重复出现，或用户再次明确要求时，才重启运行时迭代；此前保持 beta，不启动大规模合成评测。

## 6. 回退

该改动只涉及一处运行时文字。若真实任务显示它诱导过程复盘或普通总结采用错误顺序，直接撤销该段新增路由即可；备份分支继续保留完整实验设计供历史查阅，不参与当前实现。

# professional-writing skill 设计

- 日期：2026-07-13（初稿同日经初审、复审两轮修订，覆盖 P1-01~06、P2-01~06，并已对齐需求讨论的原始结论）
- 状态：已评审通过（三轮），提交于 08e85f0；本次为 plan 评审 P2-04 的术语残留修正
- 结论：新建一个专业文档写作 skill（一个公共写作流程 + 四类文档骨架），主干与用户本人思考顺序同构（读者 → 目标与中心意图 → 证据 → 叙事地图 → 章节叙事）；三道确认门 + 快速通道控制交互密度；证据优先的混合资料策略约束依据可靠性；成稿前做 fresh-context 新读者测试；交付"干净成稿 + 对话内质检摘要"。

## 1. 背景与痛点

Agent 产出的给人看的文档（调研总结、方案说明、复盘、专业文章）叙事结构不佳：结论埋得深、按执行时间线流水账、罗列无取舍、只有主题没有命题。

市面调研结论（2026-07-13，初审修正后）：已有方案覆盖共创流程或局部场景，但没有一个同时满足我们的范围、证据策略和交互习惯，故自建。可借鉴对象：

| 来源 | 借鉴点 |
| --- | --- |
| anthropics/skills doc-coauthoring | 三阶段工作流与退出条件、fresh-context reader test、模式路由（可拒绝/可跳过阶段）、按节精化循环；缺文档类型的叙事骨架路由，且流程偏重 |
| anthropics/skills internal-comms | 按文档类型分文件、识别类型后加载对应指引 |
| ComposioHQ content-research-writer | 共同定大纲、研究与引用分离、逐段反馈 |
| gemini-cli docs-writer | 以代码事实为准、主动语态、术语与格式一致性 |
| nousresearch research-paper-writing | 一句话贡献、What/Why/So What、局部与全局两轮修改 |
| pavithramailbox/Claude-ai-technical-writing-skill | 薄入口（47 行）+ reference 按需加载 + "常识而非清单"的校验哲学 |
| 本仓库 khazix-writer | 人与 AI 职责边界、章节叙事节奏、分层质检、素材不足时主动问 |
| 本仓库 technical-proposal-review | evals 区分性断言（Avoid 列表即判别器）、反馈闭环 |

构建方案对比已在需求讨论中定案：不改造 doc-coauthoring（流程偏重、且与上游同步机制冲突），不拆"路由 + 四个子 skill"（首版过度设计）；采用**新建单 skill：一个公共工作流 + 四类文档骨架**，评测证明某类文档明显分叉后再拆子 skill。

## 2. 目标 / 非目标

**目标**：让 agent（或用户显式调用时的 Claude）产出的专业文档中心意图先行（论证型结论先行、教程任务承诺先行）、按读者认知路径组织、有取舍、依据可靠；并能诊断和重写已有的结构差的文档。

**覆盖范围（四类，面向专业读者的非营销文档）**：总结与汇报（调研总结、复盘、进展汇报、变更总结）、技术解释与专业文章、方案与决策文档、教程与操作指南。

**非目标**：

- 不覆盖公众号文风模仿（khazix-writer）、营销文案、深度研究（hv-analysis / deep-research）、技术方案评审（technical-proposal-review）、交接包（handoff）、DOCX/PDF 排版（docx / pdf skill）。
- 不覆盖 API 参考/代码级文档（Diátaxis 的 reference 象限）、代码注释、聊天式简答。
- 不管句子级风格润色（可另装 obra/the-elements-of-style 互补）。
- 不做全类型模板库（internal-comms 式），只提供骨架模式 + 判词。
- 首版不加自动校验脚本；evals 证明某类机械问题（术语漂移、标题层级、空话密度）反复出现后再补。

## 3. 命名

**决定：`professional-writing`**（2026-07-13 拍板，采用宽范围）。理由：覆盖四类专业文档，与用户最初表述"专业性质的文档写作"一致。

历史备选：`engineering-narrative`（初稿窄范围命名，随范围放宽弃用）、`narrative-doc-writer`、`doc-narrative-writing`。

## 4. 触发设计（description 草案）

覆盖两个入口：用户显式请求 + agent 自主产出文档前。

- **触发词**：写总结、调研总结、写报告、复盘、复盘报告、方案说明、决策材料、汇报材料、写篇技术文章、专业文章、教程、操作指南、整理成文档、写成文档给人看、write a summary/report/postmortem/design doc/tutorial/article。
- **隐式触发**：agent 完成一段工作后要输出总结性文档给人阅读时。
- **SKIP（显式写进 description）**：公众号文章 → khazix-writer；评审已有技术方案 → technical-proposal-review；交接包 → handoff；深度研究 → hv-analysis；API 参考/代码注释；聊天式简答。
- **风险注记**：隐式触发可能误伤普通任务结束时的简短口头总结——description 中限定"产出物是给人阅读的文档文件"才触发，聊天内三五句总结不触发。

## 5. 运行时流程（SKILL.md 主体）

### 5.0 模式路由（进入任何模式前先判断）

默认共创、允许快速通道。按用户表达路由：

| 用户表达 | 路由 |
| --- | --- |
| "直接出稿" / 给全素材要结果 / agent 自主场景 | 快速通道：跳过三道门的交互确认，全部用自检替代，一次成稿，完成后统一 review |
| "先跟我讨论" / 素材或意图不明确 | 完整通道：走三道确认门 |
| "帮我改一下" / 给了已有文档 | 模式二（重写）：不重走全流程，先诊断再外科式修改 |

**交互原则（防高频盘问）**：确认门不等于连续提问。Agent 先根据材料完成判断，把假设和建议摆出来让用户确认或纠偏；只有无法可靠推断的关键问题，才一次问一个。用户中途可随时要求跳过或加密度。

### 5.1 模式一：从零写

主干采用用户本人写材料的思考顺序（读者 → 目标与中心意图 → 大纲 → 章节叙事），与用户心智模型同构，降低长期校准成本。证据地图、判词自检、新读者测试为 agent 特有补强。**默认精简、按需升级**：主流程按最高频场景（agent 自主写总结）取最短路径；读者卡、五列叙事地图等仪式只在长文、对外文章等低频重要交付时启用。**思考顺序与落笔顺序是两回事**：思考按 1→6，落笔时先写最核心最不确定的章节，标题、开头、摘要最后写。

1. **想读者**（写前必答，答不出就问用户；无人在环时优先从任务上下文推断读者，无法推断时默认为：不在场、具备领域通识但不了解本次上下文的同岗位专业读者）。主流程只回答三问：读者是谁？最关心什么？可能质疑什么？完整的六项读者卡（加阅读场景/已有认知/阅读时间）是长文、对外文章的选用模板，放 `references/narrative-patterns.md`，不作为每次必填项。

2. **定目标与中心意图**（两件事，分开定）：
   - 写作目标：读者读完后发生什么变化（知道什么、相信什么、做什么）；
   - 中心意图，按文档类型二分：
     - 论证型（总结/汇报、方案/决策、技术解释/专业文章）→ **核心命题**：一句可被反驳的断言，全文塔尖。判定：是断言不是主题——"本文介绍 X 的调研结果"是主题，"X 方案可行，建议走 B 路线"才是命题。支撑结构一并写出："这篇文档主张 X，主要依据是 A、B、C。"
     - 任务型（教程/操作指南）→ **任务承诺**：读完后读者能完成什么、前提是什么。判定：承诺可验证（有明确的完成标志），不硬造观点命题。

   这句话（命题或承诺）写不清楚，就不该列大纲。

3. **盘证据（证据地图，条件式）**：素材含外部信息、时效信息或跨来源判断时才展开——按四格盘点：事实（有出处）/ 判断（自己的推断，标注依据）/ 推测（待验证，显式标注）/ 未知（缺口）。总结自己刚完成的工作时通常跳过四格盘点仪式，但**不豁免最小事实核对**（判词 7 底线）：验证过的才写"已验证"，跑了一半的测试不能写成全过。检索规则：
   - 现有材料足够：直接写，不为了显得丰富而额外检索；
   - 关键事实可能过期且容易核实：自动查官方或一手来源；
   - 缺失信息会改变中心意图：用户在场则列出缺口征求同意后再研究；自主场景不得按原意图继续写——把命题/承诺收窄到已确认事实能支撑的范围（如从"X 方案可行"收窄为"X 在 A 场景初步可行，B 场景待确认"），收窄理由写进质检摘要；
   - 无法确认的信息：明确标记为判断、推测或待确认，不得自行补全；
   - 证据清单是写作过程的内部产物，不默认塞进最终成稿。

   → **确认门 1（写作契约）**：向用户复述"读者 + 目标 + 中心意图（命题或任务承诺）+ 事实边界（含文档类型、篇幅）"，确认后继续。快速通道下自检替代。

4. **画叙事地图**（读者认知路径，不是材料分类桶）：列出读者接受中心意图前会依次追问的问题，章节按序回答；主流程每章一行"读者的问题 → 本节结论"即可（教程按任务路径：每章一行"操作目标 → 完成标志"）；取舍在这一步做——不服务中心意图的材料砍掉或移附录。完整五列表格（加证据、认知变化两列）是长文/对外文章的选用模板，放 references。骨架模式按文档类型选起点（见 `references/narrative-patterns.md`）；标题风格按类型选：决策材料偏断言式，教程偏任务式，解释文章偏读者问题式。

   → **确认门 2（叙事地图）**：完整通道下给用户确认再动笔；快速通道下判词自检替代。

5. **写章节**：先写最核心、最不确定的章节（方案文档先写核心方案与权衡，总结先写成果与证据）。结论先行是分形的——每节首句即该节结论，节间过渡承接读者的下一个追问（教程的步骤章节例外：首句是本节操作目标，正文是步骤 + 验证，不强制结论句）；局部叙事按文档类型选（不强制每章同构）；细节只为论点服务。正文完成后，最后写标题、开头和摘要——此时才知道文章真正说了什么。（原始讨论中的七项"章节卡片"不采用：其价值已被"首句即结论 + 承接读者下一问"两条规则覆盖，立卡是仪式负担。）

   → **确认门 3（条件式代表章节）**：长文、对外文章、新文风且用户在场时，先写最关键的一章校准叙事密度与表达风格，认可后再展开全篇；短总结、常规汇报或快速通道跳过。

6. **自检**：一个环节、一份清单——`references/self-review.md` 把判词表七条、四个可操作测试、局部检查（段落清楚、转场自然）合并组织成单份清单（局部/全局只是清单内的分组），不是三套并行检查。

7. **新读者测试（fresh-context reader test，条件式）**：重要交付（用户显式要求高质量、或对外/对上材料）时执行。做法借 doc-coauthoring：预测 5-10 个读者问题 → 派一个只拿到成稿的新 subagent 作答并指出歧义、隐含前提、内部矛盾 → 按结果回步骤 5 修订。无 subagent 环境的退路：生成读者问题清单交给用户在新会话手动测试。日常自主场景不派 subagent，但至少做轻量版：自己以"读者只看得到成稿"的假设过一遍读者问题。

### 5.2 模式二：重写（治已写完的文档）

不重走全流程，先诊断再外科式修改：

1. **诊断**：对照判词表列出命中项 + 具体位置。
2. **分级处置**：局部问题（个别章节顺序、空标题、结论后置）在原文上外科式修改，不另起重复内容；结构性问题（无中心意图、整体流水账）才按读者 + 中心意图重构叙事地图后重写。
3. **收尾**：走模式一第 6-7 步（合并自检 + 条件式新读者测试）。

### 5.3 Avoid 判词表 v1（放 SKILL.md 正文，是本 skill 的灵魂）

| # | 判词 | 判定方法 |
| --- | --- | --- |
| 1 | 结论埋深或无中心意图 | 读完第一屏（前 ~10 行）仍不知道"所以呢"；论证型文档提炼不出一句可被反驳的断言（只有主题没有命题）；教程提炼不出可验证的任务承诺 |
| 2 | 执行时间线叙事 | 组织顺序是"我先做了 A 再做了 B"，而非读者关心度（教程例外：操作步骤本身就是读者要的顺序） |
| 3 | 罗列无取舍 | N 个发现/要点平铺，无主次、无权重、无"最重要的是" |
| 4 | 缺"为什么"层 | 只说做了什么，不说为何这么选、排除了什么备选 |
| 5 | 空标题 | 标题是"背景/分析/总结"式栏目名而非信息载体；检验：全部标题连读应构成完整故事线（决策/汇报类）或完整任务路径（教程类） |
| 6 | 读者错位 | 给决策者的文档塞实现细节，或给执行者的文档只有愿景 |
| 7 | 断言无依据 | 关键断言既无出处也无标注的推断依据，或把推测写成事实 |

注：判词 1-6 为通用规则（判词 1、2、5 已按论证型/任务型分别定义判定方法）；判词 7 由证据地图步骤保证。其余类型适配问题留待首战反馈校准。

## 7. references 规格

**narrative-patterns.md**：金字塔原理与 SCQA 的操作化说明；两个选用模板（六项读者卡、五列叙事地图，标注仅长文/对外文章启用）；四类文档各给"读者默认问题清单 + 推荐骨架 + 标题风格"：

- 总结/汇报：结论 → 证据 → 影响 → 风险 → 下一步；
- 方案/决策：问题 → 约束 → 选项 → 推荐 → 权衡 → 决策事项；
- 技术解释/专业文章：问题 → 核心判断 → 推理与证据 → 反例/边界 → 启示；
- 教程/操作指南：任务承诺 → 前置条件 → 步骤（每步含验证）→ 故障处理。

**evidence-map.md**：四格盘点法（事实/判断/推测/未知）操作说明、混合资料策略的检索规则、不确定性的行文标注方式。

**self-review.md**：单份合并自检清单（判词表七条 + 四个可操作测试 + 段落级局部检查，按局部/全局分组）+ 新读者测试执行步骤（预测问题清单模板、subagent 指令模板）。四个可操作测试：

- 中心意图测试：论证型能否提炼出一句可被反驳的命题，任务型能否提炼出可验证的任务承诺；
- 标题连读测试：抽出全部标题连读，应构成完整故事线或任务路径；
- 第一屏测试：只读第一屏，读者能否复述中心意图；
- 删减测试：每节问"删掉这节读者会损失什么"，答不出就删。

## 8. evals 规格

按 skill-creator 规范执行（`evals/evals.json` + workspace 按 iteration 组织），**每个用例双跑对比：with-skill vs 无 skill 基线**，证明 skill 增量而非"命中关键词"。

- 客观断言只检查可判定项：结构性事实（结论/意图是否在首段、是否有显式取舍标记、推测是否有标注、教程是否含前置条件与验证步骤）；
- 标题连读是否成故事线、读者错位等主观判词、读起来是否舒服等叙事质量由人工盲评：两版产出去标识后由用户判断优劣。

用例覆盖三种运行场景 × 四类文档骨架（每类骨架至少出现一次；首战验收同口径）：

- **eval-1（重写 × 总结/汇报)**：输入一份合成的流水账调研总结（故意埋入判词 1/2/5）。断言：诊断命中 ≥3 条判词且指出位置；重写稿结论在首段。
- **eval-2（从零写 × 方案/决策）**：输入散乱 bullet 调研笔记 + 要求"写给决策者"。断言（均可判定）：首段含一句命题式断言；有显式取舍标记（主次排序或"细节见附录"）；推断有依据标注；无"背景/分析/总结"式空标题。
- **eval-3（资料不足 × 任意类型）**：输入只有主题没有素材的请求。断言：不硬写；先问读者/目标或列出未知格缺口征求同意，再动笔。
- **eval-4（从零写 × 技术解释/专业文章）**：输入一个技术决策的散乱笔记（如"为什么放弃 X 换 Y"），要求写给技术读者的解释文章。断言：首段含核心判断；有反例/边界内容；推断有依据标注。
- **eval-5（从零写 × 教程）**：输入一段操作过程记录，要求写成操作指南。断言：开头是任务承诺而非观点命题；含前置条件与每步验证；步骤章节无强造的结论句。

内容 evals 通过后，第二阶段做**触发评测**：should-trigger / should-not-trigger 近邻用例（重点区分：正式文档请求 vs 普通任务结束的口头总结、四类范围内 vs SKIP 邻居 skill 的场景），校准 description。

## 9. 交付契约

默认"干净成稿 + 对话内质检摘要"：

- 短内容直接在对话中交付；较长或需持续维护的文档写入 Markdown 文件；
- 用户提供现有文档时，在原文基础上外科式修改，不另起重复内容；
- 写作契约、证据盘点、叙事地图默认留在对话中，不制造中间文件；
- 最终文档只保留给读者看的正文，不混入 agent 的分析过程、自检记录、判词引用；
- 交稿时在对话中附简短质检摘要：中心意图是否清楚、证据是否充分、是否通过新读者测试、还有哪些事实待确认；
- 只有用户明确要求，才另外保存大纲、研究材料或审校报告。

## 10. 构建与验证路径

1. spec 复审通过后按仓库流程提交 spec；
2. 用 skill-creator 建骨架，写 SKILL.md 首版（模式路由 + 判词表 + 两模式流程）；
3. 写 references 三文件；
4. 按 skill-creator 规范造 evals 五用例（覆盖四类骨架 × 三种运行场景），跑 with/without 对比 + 人工盲评；内容 evals 通过后做第二阶段触发评测（should/should-not-trigger 近邻用例）；
5. 部署：手工执行两条软链（已核实链路：`~/.cc-switch/skills` 是指向本仓库 `skills/` 的目录级软链，新 skill 目录建好后经它自动可见，只需给两个宿主各补一条单 skill 软链）：

   ```bash
   ln -s ~/.cc-switch/skills/professional-writing ~/.claude/skills/professional-writing
   ln -s ~/.cc-switch/skills/professional-writing ~/.codex/skills/professional-writing
   ```

   本次保持手工部署，不顺带扩展 `sync-links.sh`（2026-07-03 plan 的待办，维持 YAGNI 搁置）；
6. 首战：用户拿真实场景实测，验收样本覆盖从零写、重写、资料不足三种；
7. 按首战反馈校准（handoff/tpr 同款闭环），沉淀 feedback 记录。

注意事项（仓库既有教训）：后续任何 SKILL.md 改动必须同步检查 references/evals 旁支文件。

## 11. 风险与开放问题

- **隐式触发的可靠性**：agent 自主写总结时是否稳定触发、以及是否误伤聊天式简短总结，由第二阶段触发评测 + 首战共同验证；退路是用户显式调用重写模式。
- **判词表 v1 凭原则提炼**（无坏例子库存），首战后大概率需要校准判词的措辞、阈值与适用文档类型——这是预期内的迭代，不是缺陷。
- **交互密度**：三道门全开时可能回到"高频盘问"；模式路由、"先判断再摆假设"的交互原则与快速通道是解法，首战关注实际打断次数。
- **流程长度 vs 薄入口**：完整流程 7 步 + 三道门，已用"默认精简、按需升级"缓解（读者卡/五列地图/章节卡片降级或砍除，主流程走最短路径）；实现时仍需警惕入口膨胀，模板和骨架全部下沉 references。

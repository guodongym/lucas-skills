## 1. 背景与痛点

Agent 产出的给人看的文档（调研总结、方案说明、复盘、专业文章）叙事结构不佳：结论埋得深、按执行时间线流水账、罗列无取舍、只有主题没有命题。

市面调研结论（2026-07-13，初审修正后）：已有方案覆盖共创流程或局部场景，但没有一个同时满足我们的范围、证据策略和交互习惯，故自建。可借鉴对象：

2026-07-14 的三次 prompt-only 文风校准均未稳定通过预注册门槛：既出现已有规则未被执行、风格问题残留，也出现为改写表达而丢失条件边界。由此确认问题不只是契约措辞，而是内容起草、风格改写与事实保真缺少流程隔离；本次设计改为流程级三段式控制，不再继续堆叠提示词。

## 2. 目标 / 非目标

**目标**：让 agent（或用户显式调用时的 Claude）产出的专业文档中心意图先行（论证型结论先行、教程任务承诺先行）、按读者认知路径组织、有取舍、依据可靠，并达到统一的专业表达标准：无内部黑话和空泛套话，简洁但不机械缩短，准确、专业、清晰、易懂；不参与本项目但具备领域常识的目标读者无需额外项目上下文即可理解。同时能诊断和重写已有的结构或表达质量差的文档。

**覆盖范围（四类，面向专业读者的非营销文档）**：总结与汇报（调研总结、复盘、进展汇报、变更总结）、技术解释与专业文章、方案与决策文档、教程与操作指南。

**非目标**：

- 不覆盖公众号文风模仿（khazix-writer）、营销文案、深度研究（hv-analysis / deep-research）、技术方案评审（technical-proposal-review）、交接包（handoff）、DOCX/PDF 排版（docx / pdf skill）。
- 不覆盖 API 参考/代码级文档（Diátaxis 的 reference 象限）、代码注释、聊天式简答。
- 不做个人文风模仿，也不做文学化、营销化润色；但基础可读性风格契约属于本 skill 的目标范围，不再排除在外。
- 不拆独立 style skill；结构、证据和表达质量由同一个 `professional-writing` 流程共同负责。
- 不做全类型模板库（internal-comms 式），只提供骨架模式 + 判词。
- 首版不加自动校验脚本；evals 证明某类机械问题（术语漂移、标题层级、空话密度）反复出现后再补。

## 3. 命名

**决定：`professional-writing`**（2026-07-13 拍板，采用宽范围）。理由：覆盖四类专业文档，与用户最初表述"专业性质的文档写作"一致。

历史备选：`engineering-narrative`（初稿窄范围命名，随范围放宽弃用）、`narrative-doc-writer`、`doc-narrative-writing`。

## 4. 触发设计（description 草案）

覆盖两个入口：用户显式请求 + agent 自主产出文档前。

- **触发词**：写总结、调研总结、写报告、复盘、复盘报告、方案说明、决策材料、汇报材料、写篇技术文章、专业文章、教程、操作指南、整理成文档、写成文档给人看、write a summary/report/postmortem/design doc/tutorial/article。
- **隐式触发**：agent 完成一段工作后要输出总结性文档给人阅读时。
- **SKIP（显式写进 description）**：公众号文章 → khazix-writer；评审已有技术方案 → technical-proposal-review；交接包 → handoff；深度研究 → hv-analysis；API 参考/代码注释；聊天式简答。
- **风险注记**：隐式触发可能误伤普通任务结束时的简短口头总结——description 中限定"产出物是给人阅读的文档文件"才触发，聊天内三五句总结不触发。

## 5. 运行时流程（SKILL.md 主体）

**专业表达标准（2026-07-14 用户批准）**：这是专业文档的基础质量要求，不是个人风格模仿。成稿无内部黑话和空泛套话，简洁但不机械缩短，准确、专业、清晰、易懂；不参与本项目但具备领域常识的目标读者，无需额外项目上下文即可理解。专业不等于复杂，简洁也不等于删掉事实、条件或边界。

- 优先写具体事实，以及“主体 + 动作 + 结果”；不用口号、营销语、比喻或抽象名词替代定义和证据。
- 一段只表达一个意思，一句只承担一个主要判断或动作；删除重复结论、仪式性开头和空泛结尾，但不设置固定句长或字数阈值。
- 黑话按语义而非死词表判断：目标读者能否解释该词指谁、做什么、产生什么结果？不能时不得根据行业惯例补出定义。
- 有精确含义的标准技术术语可以保留，并在首次出现时按读者需要解释；标题中的首次出现也算首次出现。完稿必须按读者可见顺序复核每个非通用缩写的首次出现。只有 source bound 明确支持时，才把内部黑话改成具体主体、动作与结果；否则保留带引号的源词，并对照事实锁按主体、动作、对象、责任范围和强度逐项说明未定义内容；不损失决策语义时可以删除纯空话。不得把“收口”等抽象词增强为“承担”“负责”等确定性职责判断。
- 保持证据关系、认识状态、判断极性与强度：未验证、无数据、待审批或证据不足只能写为未知、待确认或证据边界；缺少证据不等于存在反证，只有 source bound 明确给出冲突事实或相反证据时才能判定冲突，反向也不得把已确认冲突弱化为待确认。

### 5.0 模式路由（进入任何模式前先判断）

默认共创、允许快速通道。按用户表达路由：

| 用户表达 | 路由 |
| --- | --- |
| "直接出稿" / 给全素材要结果 / agent 自主场景 | 快速通道：跳过三道门的交互确认，全部用自检替代，一次成稿，完成后统一 review |
| "先跟我讨论" / 素材或意图不明确 | 完整通道：走三道确认门 |
| "帮我改一下" / 给了已有文档 | 模式二（重写）：不重走全流程，先诊断再外科式修改 |

**交互原则（防高频盘问）**：确认门不等于连续提问。Agent 先根据材料完成判断，把假设和建议摆出来让用户确认或纠偏；只有无法可靠推断的关键问题，才一次问一个。用户中途可随时要求跳过或加密度。

### 5.1 模式一：从零写

主干采用用户本人写材料的思考顺序（读者 → 目标与中心意图 → 大纲 → 章节叙事），与用户心智模型同构，降低长期校准成本。证据地图、三段式质量控制、判词自检、新读者测试为 agent 特有补强。**默认精简、按需升级**：主流程按最高频场景（agent 自主写总结）取最短路径；读者卡、五列叙事地图等仪式只在长文、对外文章等低频重要交付时启用。**思考顺序与落笔顺序是两回事**：思考按 1→9，落笔时先写最核心、最不确定的章节，标题、开头、摘要最后写。

1. **想读者**（写前必答，答不出就问用户；无人在环时优先从任务上下文推断读者，无法推断时默认为：不在场、具备领域通识但不了解本次上下文的同岗位专业读者）。主流程只回答三问：读者是谁？最关心什么？可能质疑什么？完整的六项读者卡（加阅读场景/已有认知/阅读时间）是长文、对外文章的选用模板，放 `references/narrative-patterns.md`，不作为每次必填项。

2. **定目标与中心意图**（两件事，分开定）：
   - 写作目标：读者读完后发生什么变化（知道什么、相信什么、做什么）；
   - 中心意图，按文档类型二分：
     - 论证型（总结/汇报、方案/决策、技术解释/专业文章）→ **核心命题**：一句可被反驳的断言，全文塔尖。判定：是断言不是主题——"本文介绍 X 的调研结果"是主题，"X 方案可行，建议走 B 路线"才是命题。支撑结构一并写出："这篇文档主张 X，主要依据是 A、B、C。"
     - 任务型（教程/操作指南）→ **任务承诺**：读完后读者能完成什么、前提是什么。判定：承诺可验证（有明确的完成标志），不硬造观点命题。

   这句话（命题或承诺）写不清楚，就不该列大纲。

3. **盘证据（证据地图，条件式）**：素材含外部信息、时效信息或跨来源判断时才展开——按四格盘点：事实（有出处）/ 判断（自己的推断，标注依据）/ 推测（待验证，显式标注）/ 未知（缺口）。总结自己刚完成的工作时通常跳过四格盘点仪式，但**不豁免最小事实核对**（判词 7 底线）：验证过的才写"已验证"，跑了一半的测试不能写成全过。检索规则：
   - 现有材料足够：直接写，不为了显得丰富而额外检索；
   - 关键事实可能过期且容易核实：自动查官方或一手来源；
   - 缺失信息会改变中心意图：用户在场则列出缺口征求同意后再研究；自主场景不得按原意图继续写——把命题/承诺收窄到已确认事实能支撑的范围（如从"X 方案可行"收窄为"X 在 A 场景初步可行，B 场景待确认"），收窄理由写进质检摘要；
   - 无法确认的信息：明确标记为判断、推测或待确认，不得自行补全；
   - 证据清单是写作过程的内部产物，不默认塞进最终成稿。

   → **确认门 1（写作契约）**：向用户复述"读者 + 目标 + 中心意图（命题或任务承诺）+ 事实边界（含文档类型、篇幅）"，确认后继续。快速通道下自检替代。

4. **画叙事地图**（读者认知路径，不是材料分类桶）：列出读者接受中心意图前会依次追问的问题，章节按序回答；主流程每章一行"读者的问题 → 本节结论"即可（教程按任务路径：每章一行"操作目标 → 完成标志"）；取舍在这一步做——不服务中心意图的材料砍掉或移附录。完整五列表格（加证据、认知变化两列）是长文/对外文章的选用模板，放 references。骨架模式按文档类型选起点（见 `references/narrative-patterns.md`）；标题风格按类型选：决策材料偏断言式，教程偏任务式，解释文章偏读者问题式。

   → **确认门 2（叙事地图）**：完整通道下给用户确认再动笔；快速通道下判词自检替代。

5. **写内容初稿并形成事实锁（阶段 A）**：内容初稿先解决结构、论证和事实完整性，暂不把“更短”当目标。结论先行是分形的——每节首句即该节结论，节间过渡承接读者的下一个追问（教程步骤章节例外：首句是本节操作目标，正文是步骤 + 验证，不强制结论句）；局部叙事按文档类型选，不强制每章同构；正文完成后再写标题、开头和摘要。

   教程逐个关键状态转换按“前态 → 使转换发生的必要动作 → 后态 → 验证证据”成文，动作与验证不能互相替代。起草前必须形成内部状态转换表，每个关键转换各占一行；source bound 未提供必要动作时，该行必须把动作记为未知，并在正文中将该转换单列为待确认步骤或阻塞项。不得省略、按惯例补造，或仅用后态和后续成功信号代替动作。

   在从零写场景，优先根据原材料和已核实证据在初稿前形成内部事实锁；已有初稿或先完成初稿的场景，可在初稿后、风格校准前补建，但初稿只能作为覆盖检查的索引，不能成为字段值的唯一来源。若用户提供的原文就是唯一原材料，source locator 指向该原文的精确位置，而不是改写后的初稿。事实锁复用 evidence map 的事实/判断/推测/未知概念；最小字段如下：

   | 字段 | 约束 |
   | --- | --- |
   | stable ID | 本次交付内稳定且唯一，如 `F-01`、`J-01`；每个 ID 只锁一个原子命题。定义、动作、职责、因果、目标或阈值是不同语义关系，必须拆开，不能因出现在同一句源文中合并 |
   | type | `事实 / 判断 / 推测 / 未知` 之一 |
   | 精确命题 | 写出必须保持不变的单一语义关系，不只记主题词；若一句同时包含“X 是什么”和“X 做什么”，至少拆成两个 ID |
   | 限定信息 | 逐项记录数值、成立条件、适用范围和否定/排他边界；没有时明确记“无” |
   | 标准术语/标识符 | 记录需要精确保留的标准术语、缩写、对象名、接口名等；没有时明确记“无” |
   | source locator | 指向原材料或已核实证据的精确位置，如文件与行号、章节、日志片段或测试结果 |
   | 支持方式 | `direct / inference / unknown` 之一。direct 表示 source locator 完整支持命题的关系和强度；inference 另记依据与 status（如建议、待决、待确认）；unknown 表示来源不足以形成结论 |

   source bound 中的黑话或含糊措辞无法确定具体主体、动作、对象、责任范围或强度时，不得把该关系混入相邻事实并标成 direct。事实锁的精确命题保留带引号的源文原词，支持方式记为 unknown，不写入行业惯例推断；同时增加“未知”项，按五个维度逐项记录未定义内容和缺失信息。可确定部分与未知部分分开锁定。

   事实锁与 source bound 在阶段 B 前一起冻结，阶段 B/C 不得暗改或引入 bound 外事实；需要新证据时回到阶段 A 更新后重新冻结。它们只存在于当前上下文，不默认落盘，也不进入成稿；即使跳过完整四格证据盘点，也不能跳过最小事实锁。

6. **独立风格校准（阶段 B）**：内容初稿完成后单独执行一次风格校准，不与内容起草混在同一轮。逐项检查并局部改写：
   - 优先使用“主体 + 动作 + 结果”；
   - 标准术语和非通用缩写首次出现时按读者需要解释，标题中的出现同样计入首次；完稿后按读者可见顺序逐个复核；
   - 一段只表达一个意思，一句只承担一个主要判断或动作；
   - 保持证据关系、认识状态、判断极性与强度；缺证不等于反证，未验证、无数据、待审批或证据不足只能表达为未知、待确认或证据边界，只有明确的冲突事实或相反证据才能支持冲突判断，已确认冲突也不得反向弱化；
   - 按“主体 + 判断/动作 + 关键条件/范围边界”扫描语义重复；每个完整命题只设一个权威展开位置，其他位置只有承担不同读者功能时才保留并压缩为引用或指向；删除仪式性开头和空泛结尾，但不机械压缩篇幅；
   - 只有 source bound 明确支持时才把内部黑话改成具体主体、动作与结果；否则保留带引号源词，并对照事实锁按主体、动作、对象、责任范围和强度逐项说明未定义内容，不损失决策语义时可删除纯空话；不得按行业惯例增强主体职责、动作、对象、范围或强度。

   → **确认门 3（条件式代表章节）**：长文、对外文章、新文风且用户在场时，先对最关键一章完成内容初稿与风格校准，确认叙事密度和表达后再展开全篇；短总结、常规汇报或快速通道跳过交互确认。

7. **对照事实锁复核（阶段 C）**：用结构化拒绝门做双向核对，不能凭通读印象声明通过。内部复核记录必须包含四个槽位：

   - **正向事实槽**：逐 atomic stable ID 记录成稿位置，确认单一精确命题、限定信息、术语/标识符、支持方式、依据和 status 均完整保留；
   - **反向主张槽**：按成稿顺序逐句拆出事实与判断，每个主张记录“原子命题 + `direct / inference / unknown` + source locator/依据”。direct 只有在 locator 完整支持该命题的主体、关系、条件和强度时成立；若 locator 只是若干相关事实，不能证明“越过目标、满足约束、导致、负责”等合成关系。inference 必须在成稿中显式标为判断/推测并写出依据与 status；unknown 不得写成成稿结论；
   - **可见顺序缩写槽**：逐个记录非通用缩写的首次可见位置及释义位置，标题计入阅读顺序；释义滞后即 mismatch；
   - **类型专项槽**：保留黑话逐项对照事实锁的主体、动作、对象、责任范围和强度未知维度；教程逐行核对状态转换表；重写逐项核对七条判词扫描结果。

   任一适用槽位缺失、存在未追溯主张、support mode 与来源不符，或教程状态转换表存在未在正文披露的未知动作，都不得声明 `0 mismatch`。只有 source locator 没有支持方式和语义蕴含核对，不算完成反向主张槽。

   逐项核对“源文黑话 → 风格稿表达”映射：动词变得更具体或更强，或者主体、对象、责任强度、范围发生变化，均算 mismatch。判断依据中的证据关系、认识状态、极性或强度变化同样算 mismatch，包括把未知、缺证或待决写成冲突、证伪，也包括把已确认冲突弱化为待确认。

   任何遗漏、弱化、扩大、缩小、改变或锁外新增都算 mismatch，最终只接受 `0 mismatch`。

   对同一问题句段只允许一次局部修复：先恢复事实锁中的原事实，只改问题所在句段，再执行双向核对；若仍 mismatch，撤销该句段的全部 style 改动，恢复 pre-style、事实正确的原文，并停止继续改写该句段。不得不加定位地盲目重写整篇；恢复原文后仍存在的风格问题写入对话内质检摘要。

   重要交付（用户显式要求高质量，或对外/对上材料）优先创建不继承当前对话或任务历史的 fresh subagent，向其提供目标读者、内容初稿、冻结事实锁、source bound 和阶段 B/C 标准，由其完成风格校准和事实复核；主 agent 只接收事实锁无 mismatch 的稿件。无 subagent 环境，或日常文档，由同一 agent 明确分开内容初稿、风格校准和事实复核三个阶段执行。快速通道只跳过交互确认，不跳过事实锁或阶段 C。

8. **合并自检**：一个环节、一份清单——`references/self-review.md` 把判词表七条、四个可操作测试、阶段 B 风格检查和阶段 C 结构化拒绝门合并组织成单份清单，不是多套并行检查。质检摘要只能报告实际完成的槽位；任何槽位未完成或失败时必须披露问题，不得写“全部覆盖”“未新增事实”或 `0 mismatch`。

9. **新读者测试（fresh-context reader test，条件式）**：重要交付时预测 5-10 个读者问题 → 派一个只拿到成稿的新 subagent 作答并指出歧义、隐含前提、内部矛盾 → 按结果局部修订。若引入新证据，或 reader test 导致事实、判断、条件、范围等内容变化，必须回阶段 A 更新并重新冻结事实锁，再执行 B/C；纯表达变化不改锁，只需再次执行阶段 C。无 subagent 环境的退路：生成读者问题清单交给用户在新会话手动测试。日常自主场景至少由作者以“读者只看得到成稿”的假设过一遍读者问题。

### 5.2 模式二：重写（治已写完的文档）

不重走全流程，先诊断再外科式修改：

1. **诊断**：先在内部对判词表 1 至 7 逐项记录“命中位置”或“未命中”，不得只扫描最显眼的问题；交付的诊断只列命中项及具体位置，不暴露未命中清单。
2. **分级处置**：局部问题（个别章节顺序、空标题、结论后置）在原文上外科式修改，不另起重复内容；结构性问题（无中心意图、整体流水账）才按读者 + 中心意图重构叙事地图后重写。
3. **收尾**：把已有文档或结构重写稿视为内容初稿，从原材料和已核实依据形成事实锁，依次执行模式一阶段 B、阶段 C、合并自检和条件式新读者测试。局部表达问题也必须经过事实复核。

### 5.3 Avoid 判词表 v1（放 SKILL.md 正文，是本 skill 的灵魂）

| # | 判词 | 判定方法 |
| --- | --- | --- |
| 1 | 结论埋深或无中心意图 | 读完第一屏（前 ~10 行）仍不知道"所以呢"；论证型文档提炼不出一句可被反驳的断言（只有主题没有命题）；教程提炼不出可验证的任务承诺 |
| 2 | 执行时间线叙事 | 组织顺序是"我先做了 A 再做了 B"，而非读者关心度（教程例外：操作步骤本身就是读者要的顺序） |
| 3 | 罗列无取舍 | N 个发现/要点平铺，无主次、无权重、无"最重要的是" |
| 4 | 缺"为什么"层 | 只说做了什么，不说为何这么选、排除了什么备选 |
| 5 | 空标题 | 标题是"背景/分析/总结"式栏目名而非信息载体；检验：全部标题连读应构成完整故事线（决策/汇报类）或完整任务路径（教程类） |
| 6 | 读者错位 | 给决策者的文档塞实现细节，或给执行者的文档只有愿景 |
| 7 | 断言无依据 | 关键断言既无出处也无标注的推断依据，或把推测写成事实 |

注：判词 1-6 为通用规则（判词 1、2、5 已按论证型/任务型分别定义判定方法）；判词 7 由证据地图步骤保证。其余类型适配问题留待首战反馈校准。

## 7. references 规格

**narrative-patterns.md**：金字塔原理与 SCQA 的操作化说明；两个选用模板（六项读者卡、五列叙事地图，标注仅长文/对外文章启用）；四类文档各给"读者默认问题清单 + 推荐骨架 + 标题风格"：

- 总结/汇报：结论 → 证据 → 影响 → 风险 → 下一步；
- 方案/决策：问题 → 约束 → 选项 → 推荐 → 权衡 → 决策事项；
- 技术解释/专业文章：问题 → 核心判断 → 推理与证据 → 反例/边界 → 启示；
- 教程/操作指南：任务承诺 → 前置条件 → 步骤（每步含验证）→ 故障处理。

**evidence-map.md**：四格盘点法（事实/判断/推测/未知）操作说明、混合资料策略的检索规则、不确定性的行文标注方式。

**self-review.md**：单份合并自检清单（判词表七条 + 四个可操作测试 + 段落级局部检查，按局部/全局分组）+ 新读者测试执行步骤（预测问题清单模板、subagent 指令模板）。局部检查区分证据不足与相反证据；阶段 B 检查“主体 + 动作 + 结果”、术语首次解释（含标题）、一段一意/一句一主判断、语义去重、认识状态/极性/强度，以及 source-bound 条件下的黑话改写；阶段 C 使用正向事实、反向主张、可见顺序缩写和类型专项四个必填槽位。事实锁与正向事实槽按原子命题工作，定义、动作、职责、因果、目标和阈值不得混入同一 ID。反向主张槽逐句记录原子命题、`direct / inference / unknown` 和 source locator/依据；locator 只有完整支持关系和强度时才算 direct，相关事实的罗列不能自动支持合成判断。重写模式先逐项扫描七条判词；教程用状态转换表逐个核对“前态 → 必要动作 → 后态 → 验证证据”，最终验证必须确认运行版本或构建标识，不得用对象存在、启动成功、健康检查或后续业务成功替代采用、加载或生效证据。检查只约束可读性与保真，不用固定句长、字数或词表替代判断；同一问题句段一次局部修复仍 mismatch 时，恢复 pre-style、事实正确的原文并停止改写该句段；只有四槽完整且无失败时才可声明 `0 mismatch`。四个可操作测试：

- 中心意图测试：论证型能否提炼出一句可被反驳的命题，任务型能否提炼出可验证的任务承诺；
- 标题连读测试：抽出全部标题连读，应构成完整故事线或任务路径；
- 第一屏测试：只读第一屏，读者能否复述中心意图；
- 删减测试：每节问"删掉这节读者会损失什么"，答不出就删。

## 8. evals 规格

按 skill-creator 规范执行（`evals/evals.json` + workspace 按 iteration 组织）。新建 skill 的首轮用例双跑 **with-skill vs 无 skill 基线**，证明 skill 增量而非“命中关键词”；修改既有 skill 时遵循 RED → GREEN → 回归。eval-6 已有 old/no-guidance RED 证据保留为本次流程升级的基线，不因设计变化重写历史结果。

- 客观断言只检查可判定项：结构性事实（结论/意图是否在首段、是否有显式取舍标记、推测是否有标注、教程是否含前置条件与验证步骤）和 fixture 预先枚举的表达事实；“明显空话”“是否足够具体”等依赖语境的判断不进入硬断言，但 fixture 预先枚举的未定义抽象表述是否被增强为具体职责属于可判定的保真断言；
- 标题连读是否成故事线、读者错位、是否简洁专业清楚、术语解释是否恰到好处等主观质量由用户查看 viewer 后最终判断，不能用固定词表、句长或字数阈值替代读者判断。

用例覆盖三种运行场景 × 四类文档骨架（每类骨架至少出现一次；首战验收同口径）：

- **eval-1（重写 × 总结/汇报)**：输入一份合成的流水账调研总结（故意埋入判词 1/2/5）。断言：诊断命中 ≥3 条判词且指出位置；重写稿结论在首段。
- **eval-2（从零写 × 方案/决策）**：输入散乱 bullet 调研笔记 + 要求"写给决策者"。断言（均可判定）：首段含一句命题式断言；有显式取舍标记（主次排序或"细节见附录"）；推断有依据标注；无"背景/分析/总结"式空标题。
- **eval-3（资料不足 × 任意类型）**：输入只有主题没有素材的请求。断言：不硬写；先问读者/目标或列出未知格缺口征求同意，再动笔。
- **eval-4（从零写 × 技术解释/专业文章）**：输入一个技术决策的散乱笔记（如"为什么放弃 X 换 Y"），要求写给技术读者的解释文章。断言：首段含核心判断；有反例/边界内容；推断有依据标注。
- **eval-5（从零写 × 教程）**：输入一段操作过程记录，要求写成操作指南。断言：开头是任务承诺而非观点命题；含前置条件与每步验证；“已上传且校验通过的发布包 → 后续启动对象实际采用目标发布包”的必要动作必须存在，素材未提供时单列待确认步骤/阻塞项；服务启动后必须核对运行版本或构建标识与目标一致，`Application started`、健康检查和业务成功不能替代版本身份验证；步骤章节无强造的结论句。促成状态变化的动作与确认结果的验证分成两条独立断言。
- **eval-6（重写 × 风格压力）**：输入一份黑话密集、非通用缩写未解释、长句和空泛表达明显，但包含必须保留的技术事实与标准术语的专业材料。fixture 在运行前枚举：指定非通用缩写清单、必须保留的事实 F1…Fn（含各自数值与成立条件）、指定重复命题，以及带明确主体但动作、对象、责任范围或强度未定义的抽象表述 U1…Un。硬断言检查：指定缩写已解释或删除；F1…Fn 的事实、数值与条件全部保留且不矛盾；指定重复命题只完整展开一次，后续引用不与其矛盾；U1…Un 只能在不损失已知事实时删除纯空话，或保留带引号源词并说明未定义内容，不得增强为更具体或更强的主体职责、动作、对象、范围或确定性判断。四项风格问题由固定 style control 检查；整体简洁性、专业度、清楚程度和易懂程度由用户查看 viewer 后判断，不以成稿更短作为成功标准。

  | --- | --- |

## 9. 交付契约

默认"干净成稿 + 对话内质检摘要"：

- 短内容直接在对话中交付；较长或需持续维护的文档写入 Markdown 文件；
- 用户提供现有文档时，在原文基础上外科式修改，不另起重复内容；
- 写作契约、证据盘点、叙事地图默认留在对话中，不制造中间文件；
- 事实锁是内部过程产物，不默认落盘、不进入成稿，也不作为独立附件交付；
- 最终文档只保留给读者看的正文，不混入 agent 的分析过程、自检记录、判词引用；
- 交稿时在对话中附简短质检摘要：中心意图是否清楚、证据是否充分、是否通过新读者测试、还有哪些事实待确认；
- 只有用户明确要求，才另外保存大纲、研究材料或审校报告；不默认生成 style report。

## 11. 风险与开放问题

- **隐式触发的可靠性**：agent 自主写总结时是否稳定触发、以及是否误伤聊天式简短总结，由第二阶段触发评测 + 首战共同验证；退路是用户显式调用重写模式。
- **判词表 v1 凭原则提炼**（无坏例子库存），首战后大概率需要校准判词的措辞、阈值与适用文档类型——这是预期内的迭代，不是缺陷。
- **交互密度**：三道门全开时可能回到"高频盘问"；模式路由、"先判断再摆假设"的交互原则与快速通道是解法，首战关注实际打断次数。
- **流程长度 vs 薄入口**：完整流程 9 步 + 三道门，已用“默认精简、按需升级”缓解（读者卡/五列地图降级，快速通道不增加交互）；实现时仍需警惕入口膨胀，事实锁只保留最小清单，模板和骨架继续下沉现有 references。
- **黑话边界依赖语境**：同一个词可能是有精确含义的技术术语，也可能是空泛套话；用目标读者语义测试和首次解释处理，不维护易误伤的禁词表。
- **prompt-only 规则遵循不稳定**：此前三次 wording 调整都未稳定通过门槛；已出现规则存在但输出仍重复空话，以及最近一轮仅 `28/30`、丢失排他条件边界的结果。流程必须把风格校准与事实复核分离，不能再假设一句提示能同时保证可读性和保真。
- **作者自证不可靠**：Batch 9 多份输出在实际存在 mismatch 或状态跳步时仍声明 `0 mismatch`。阶段 C 因此必须留下四个内部验收槽位；缺槽等同失败，质检摘要不得用结论性措辞代替逐项检查。
- **locator 不是支持本身**：Batch 10 的 eval-4 为合成判断列出了相关原始条目，但条目没有提供重建和扩容的目标阈值。反向主张槽必须判断来源是否蕴含主张的完整关系和强度；只列 locator 不得通过。
- **简洁可能损伤精度**：机械压缩容易删掉依据、条件和边界；事实锁 mismatch 直接拒收风格稿，恢复原事实后局部重写。本轮 7 份 current-skill 文档同时检查可读性和内容完整性，其中 eval-2、eval-4 重点检查证据与技术精度，不以更短作为单独成功标准。
- **judge construct mismatch**：已有通用 blind comparator 会把结构偏好与四项风格问题混合，并可能不对称处罚相同问题；本轮不运行 A/B，流程门只使用固定事实硬断言、同四项绝对 style control 和用户对 viewer 的人工判断。

# Feedback Loop

Every review response should include a feedback block. The user can fill it in directly or provide short natural language feedback such as "P1-02 喜欢，P2-03 太泛，漏了迁移回滚".

## Feedback Block

```yaml
case_id: 2026-04-xx-short-name
proposal_path:
source_scope:
  reviewed_input:
  existing_review_comments:
  evidence_source:
skill_review_version:
skill_result: pass | revise | block | human_focus
human_final_result: pass | revise | reject | pending

review_quality_feedback:
  missed_issues:
    - issue:
      severity:
      should_have_matched_rule:
  false_positives:
    - skill_comment_id:
      reason: not_relevant | too_strict | already_covered | wrong_context
  severity_adjustments:
    - skill_comment_id:
      from:
      to:
      reason:

output_preference_feedback:
  liked_comments:
    - skill_comment_id:
      reason: accurate | important | actionable | matches_my_style
  disliked_comments:
    - skill_comment_id:
      reason: too_generic | too_verbose | not_actionable | wrong_tone | low_priority
  wording_preferences:
    - before:
      after:
  reviewer_notes:
```

## Processing Rules

- Missed issues become candidates for `risk-patterns.md`.
- False positives tighten evidence requirements or reduce severity.
- Severity adjustments refine P0/P1/P2 boundaries.
- Liked comments inform future wording and prioritization.
- Disliked comments become output anti-patterns in `output-preferences.md`.
- Feedback generated by bootstrap replay, model self-review, or another agent must first be stored in `feedback/pending/`.
- Only human-approved feedback can move to `feedback/accepted/` and be folded into `risk-patterns.md`, `case-bank.md`, or `output-preferences.md`.
- Accepted feedback can be folded into references.
- Rejected feedback stays traceable and does not modify behavior.

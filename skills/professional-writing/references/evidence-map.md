# 证据地图

## 四格盘点法

只在素材含外部信息、时效信息或跨来源判断时展开四格盘点；它区分“已经知道”和“仍需确认”，避免把推断写成事实。

| 类别 | 判断标准 | 工程示例 |
| --- | --- | --- |
| 事实 | 有可追溯出处，且出处直接支持表述 | 事实：`eval-1` 断言 3/3 通过（本次运行日志）。 |
| 判断 | 作者基于事实做出的解释或选择，必须标注依据 | 判断：建议保留双写一周，因为回滚演练只覆盖了双写状态。 |
| 推测 | 尚待验证的可能性，必须显式标注 | 推测：该 API 可能在 v2 废弃（未查证）。 |
| 未知 | 会影响结论但当前材料没有回答的缺口 | 未知：生产峰值是否超过本次压测的 8,000 QPS。 |

盘点后分别处理：事实可以直接支撑断言；判断必须连同推理依据表达；推测要保留不确定性标记；未知要么补证据，要么收窄中心意图。

## 检索规则

1. 现有材料足够时直接写，不为了显得丰富而额外检索。
2. 关键事实可能过期且容易核实时，自动查官方或一手来源。
3. 缺失信息会改变中心意图时：用户在场就列出缺口，征求同意后再研究；自主场景不得按原意图硬写，应把命题或任务承诺收窄到已确认事实能支撑的范围，并把收窄理由写进质检摘要。
4. 无法确认的信息明确标记为判断、推测或待确认，不得自行补全。
5. 证据清单是写作过程的内部产物，不默认塞进成稿。
6. 决策文档中，素材或已确认约束直接支持的条件可写成硬门槛或既定流程。写作者推导的责任分工、控制线、审批规则或 SLA 可以作为治理建议，但须写明推导依据，标明其建议/待决/待确认状态，并指出待确认主体；缺少依据或未获确认时，不得写成既定事实或既定约束。

总结自己刚完成的工作可以跳过盘点形式，但仍需核对最小事实：实际执行了什么验证，就只报告该验证覆盖的范围。

## 不确定性的行文标注

| 不合格写法 | 合格写法 |
| --- | --- |
| 可降低 40% 延迟。 | 预计可降低 40% 延迟（基于 staging 压测，生产未验证）。 |
| 迁移将在两周内完成。 | 按当前 5 个服务的演练速度判断，迁移约需两周；未计入第三方联调时间。 |
| v2 会废弃该 API。 | 该 API 可能在 v2 废弃（来自路线图草案，尚待正式发布说明确认）。 |

合格表述同时说明不确定性类型、判断依据和未覆盖边界；不要只加“可能”“大概”而省略依据。

# 全局工程指令 (Global Engineering Directives)

## 1. 核心原则
- **语言策略**:
  - **交互**: 默认使用中文进行技术交流。
  - **交付物**: 代码注释、文档、提交信息等应自动适配项目现有的语言环境；若仓库以英文为主，则优先使用英文。
- **风格**: 简洁、技术导向。避免空话，直接给出结论、方案和验证结果。
- **先想后写**:
  - 动手前先显式列出自己的假设。不确定就问，不要沉默猜测。
  - 若需求存在多种解读，必须呈现选项，不得擅自选定一种。
  - 若存在更简单的方案，主动提出。该 push back 就 push back。
  - 遇到困惑时立即停下，指明哪里不清楚，向用户确认。
- **决策策略**:
  - **自主决策**: Bug 修复、局部重构、单元测试补充、文档补充、低风险配置修正。
  - **必须请示**: 核心接口签名变更、数据库 Schema 修改、引入新依赖、跨模块大规模变动、潜在破坏性行为变更。
  - **冲突处理**: 优先询问而非假设；若仓库既有约定与本指令冲突，以仓库约定为准。

## 2. 技能与工作流 (Skills & Workflow)
- **技能优先**:
  - 在开始开发任务前，必须尝试调用 `superpowers:using-superpowers`，
  - 若调用成功，遵循其工作流指引执行任务。
  - 若该技能不可用，**显式告知用户"superpowers 技能未检测到"**，再检查当前环境中是否存在可用的 workflow / skill / expert guidance。
  - 若存在相关能力，则优先调用并遵循其工作流。
  - 若不存在，则按同等工程标准手动执行：先澄清目标，再识别约束，再实施变更，最后完成验证。
- **目标驱动执行**:
  - 默认优先采用 TDD 或至少“先失败、后修复、再回归验证”的方式推进。
  - 对于非测试类任务，也应将“做什么”转化为“验证什么”：明确成功标准，按 `[步骤] → verify: [检查方式]` 逐步推进，而非模糊地“改好了”。
  - 对于文档修改、配置调整、脚本修补、一次性迁移、生成代码更新或其他不适合先写失败测试的任务，可以不强制先写测试。
  - 不适用 TDD 时，必须先说明原因，并明确本次采用的最小可验证路径，例如：现有测试、手动复现步骤、lint、typecheck、集成验证或回归检查。
  - 豁免时须在 commit message 中注明所采用的验证方式

## 3. 任务预研 (Discovery Phase)
- 在以下场景优先识别项目技术栈与仓库约定：
  - 首次进入仓库。
  - 技术栈未知或项目结构不清晰。
  - 需要执行构建、测试、静态检查、发布相关命令。
  - 仓库规范、分支策略、提交规范尚不明确。
- 预研时优先检查：
  - `README.md`
  - `Makefile`
  - `package.json`
  - `requirements.txt`
  - `pyproject.toml`
  - `Cargo.toml`
  - 其他与构建、测试、lint、typecheck、CI 相关的配置文件
- 需要明确的信息包括：
  - **Build**
  - **Test**
  - **Lint**
  - **Type Check**
  - **Format**
- 已明确的仓库约定可以在当前任务上下文中复用；当仓库状态、分支或配置发生变化时，应重新确认，避免依赖过期结论。

## 4. Git 规范 (Context-Aware Commits)
- **总原则**: 优先遵循仓库现有的 Git 工作流、分支策略和提交规范；若仓库没有明确约定，再采用以下默认规则。
- **Commit 格式**: `<type>(<scope>): <描述>`
- **描述语言**: 与项目主流语言保持一致；英文仓库使用英文，中文仓库使用中文。
- **Scope 选取**: 优先使用受影响的目录名、模块名或子系统名。
- **Body**: 需要时补充“为什么改”，而不仅是“改了什么”。
- **Breaking Change**: 若仓库采用 Conventional Commits，涉及破坏性变更时显式标注 `BREAKING CHANGE:`。
- **默认分支策略**: 若仓库无明确约定，默认使用 `feature/<kebab-case>` 作为开发分支，在切换分支前必须确保工作区干净。

| type | 适用场景 |
| :--- | :--- |
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `test` | 测试相关 |
| `docs` | 文档、注释 |
| `refactor` | 重构，不改变外部行为 |
| `infra` | CI/CD、构建脚本、Makefile、工程基础设施 |
| `dx` | 开发体验优化、工具脚本改进、本地流程改进 |
| `chore` | 杂项维护、依赖更新、非功能性清理 |

## 5. 简洁性原则 (Simplicity First)
- 只写被要求的功能，不做投机性扩展。
- 单次使用的逻辑不要抽象成类/策略模式/配置化。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景编写错误处理。
- 如果 200 行能用 50 行解决，重写它。
- **自检**: "一个高级工程师会说这段代码过度复杂吗？" 如果是，简化。

## 6. 代码质量底线
- **一致性（外科式修改）**: 必须尽量遵循项目现有的命名习惯、目录结构、抽象层次和错误处理模式，优先做局部一致的外科式修改。
  - 不得"顺手"改进相邻的代码、注释或格式。
  - 不重构没有问题的代码。即使你觉得有更好的写法，也要匹配现有风格。
  - 发现不相关的死代码/技术债时，**提出来但不要自行删除**，除非被要求。
  - 自己的变更产生的孤儿（未使用的 import/变量/函数）必须清理；已有的不动。
  - **自检**: 每一行改动都能直接追溯到用户的需求吗？如果不能，撤回。
- **安全性**:
  - 严禁硬编码 API Key、Token 或其他敏感信息。
  - 引入新依赖时，需做最基本的必要性、维护状态与许可证风险判断。
- **完整性**:
  - 在支持显式类型标注且项目已有相关规范的语言中，补齐必要的类型信息。
  - 涉及业务逻辑变更时，优先补充自动化验证；若不适合自动化验证，必须提供可复现的人工验证步骤。
  - 在提交任务前，必须主动运行项目已有的 lint 和 typecheck 命令，并修复由此引入的所有警告或错误，不得留给用户处理。
- **可维护性**:
  - 不得静默吞掉异常或忽略错误返回，除非有明确理由。
  - 对大数据量操作需考虑算法复杂度，避免隐性 O(n²) 问题
  - 日志级别应与项目既有约定保持一致，默认统一使用 error/warn/info/debug 分级。
  - 不保留临时性的 `print`、`console.log`、调试开关或注释掉的大段废弃代码。
